SunyaSuite — System Architecture Diagrams

Academic-grade UML diagrams for project synopsis

Figure 1 — Clean Architecture Layer Diagram (UML Package Diagram Notation)

Figure 1. Clean Architecture layer diagram in UML package notation. The four layers follow the Dependency Inversion Principle: outer layers (presentation, infrastructure) depend on abstractions defined in inner layers (application, domain). No layer has knowledge of layers outside itself. The Domain layer at the core contains entities, enums, and constants with zero external library dependencies beyond the .NET runtime. Each project name is annotated in monospace.

Figure 2 — Deployment Architecture (UML Deployment Diagram)

Figure 2. Three-tier deployment architecture in UML deployment diagram notation. The client workstation runs a standard web browser connected via HTTPS to a reverse proxy (nginx/Caddy) that handles SSL termination. The application server runs Kestrel with Blazor Server interactive rendering and the REST API. The database server hosts PostgreSQL 16 with both the shared config database and per-organization tenant databases. Connections between tiers use secure network protocols.

Figure 3 — Invoice Creation Workflow (UML Activity Diagram with Swimlanes)

Figure 3. UML activity diagram with swimlane partitions showing the invoice creation workflow. The User lane contains only the form-filling step; all other actions execute on the server side in the System lane. The decision node checks whether the active fiscal year is open — if not, an error is returned immediately. After successful creation, the workflow forks into parallel actions: showing a success toast notification and redirecting to the invoice detail page.

Figure 4 — Entity-Relationship Diagram (Crow’s Foot Notation)

Figure 4. Entity-Relationship diagram in Crow's Foot notation. The config schema (top) contains three tables: Organization, ApplicationUser, and the associative entity OrganizationUser (carrying a Role discriminator). The tenant schema (bottom) contains the domain entities. Primary keys are annotated with PK, foreign keys with FK. One-to-many relationships are shown with Crow's Foot symbols — the "1" end has a single line, the "many" end shows the characteristic three-pronged foot. Key relationships: Company to Branch (1:*), Client to Invoice (1:*), Invoice to InvoiceItem (1:*), and the many-to-many between Invoice and MoneyReceipt mediated through ReceiptInvoiceAllocation.

Figure 5 — Use Case Diagram (UML Notation)

Figure 5. Use case diagram showing the three actor types and their permitted system functions. Org Members access client, project, invoice, and receipt management, plus PDF generation and Excel export. Org Admins additionally manage companies, branches, and fiscal years. System Admins manage organizations, users, and invites at the global level. Generalization relationships indicate that Org Admin inherits all use cases of Org Member, and System Admin is a separate role hierarchy. The «include» relationship from Manage Invoices to Generate PDF indicates that PDF generation is an included sub-functionality of invoice management.

Figure 6 — Core Class Diagram (UML Notation)

Figure 6. Core class diagram in UML notation showing the principal domain entities and their relationships. Each class is displayed in a three-compartment box: name (with stereotype), attribute list with visibility markers (+ for public), and operations or constraints. Navigability is indicated by association arrows. The multiplicity annotations show one-to-many relationships (1 — *). ApplicationUser inherits from ASP.NET Core IdentityUser. Invoice has a public Recalculate() operation that recomputes totals from line items. InvoiceItem's Amount property is computed (derived) from Quantity × UnitPrice. ClientStatus is a defined enumeration with values Green, Yellow, and Red.
